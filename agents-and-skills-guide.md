# คู่มือการใช้งาน Cursor Skills และ Agents

เอกสารนี้อธิบายวิธีใช้งาน `skills` และ `agents` ภายใน `.cursor/` เพื่อให้การทำงานของ AI มีมาตรฐาน สอดคล้องกับบริบทของโปรเจกต์ และช่วยให้เลือกเครื่องมือที่เหมาะสมกับงานได้เร็วขึ้น

---

## 1) ภาพรวมของระบบใน `.cursor/`

โดยทั่วไปโฟลเดอร์ `.cursor/` จะมีองค์ประกอบหลัก ๆ ดังนี้

- `rules/` — กฎถาวรที่ใช้ควบคุมพฤติกรรมของ agent
- `skills/` — ความสามารถเฉพาะทาง แยกตามงานหรือโดเมน
- `mcp.json` — การตั้งค่าบริการหรือเครื่องมือเสริม
- ไฟล์ประกอบอื่น ๆ เช่น ตัวอย่าง, reference, หรือเอกสารช่วยใช้งาน

แนวคิดสำคัญคือ

- **rules** ใช้กำหนด “มาตรฐานการทำงาน”
- **skills** ใช้กำหนด “ความสามารถเฉพาะทาง”
- **agents** คือรูปแบบการมอบหมายงานให้โมเดล/ผู้ช่วยทำงานตามบทบาทที่ชัดเจน

---

## 2) หลักการเลือกใช้ `rules`, `skills`, และ `agents`

### ใช้ `rules` เมื่อ

- ต้องการให้พฤติกรรมบางอย่างเกิดขึ้นเสมอ
- ต้องการกำหนดมาตรฐานข้ามทุกโปรเจกต์หรือทุก session
- ต้องการข้อห้ามหรือข้อปฏิบัติที่ชัดเจน เช่น
  - ห้ามเดา API
  - ต้องอ่านไฟล์ก่อนแก้
  - ต้องเช็ก lint/test หลังแก้โค้ด

### ใช้ `skills` เมื่อ

- งานมีรูปแบบซ้ำ ๆ และต้องใช้วิธีเฉพาะทาง
- ต้องการให้ agent ทำงานได้แม่นขึ้นตามประเภทงาน
- ต้องการแยกบทบาท เช่น
  - วิเคราะห์ codebase
  - เขียนโค้ด
  - debug
  - review
  - test
  - deploy
  - docs

### ใช้ `agents` เมื่อ

- งานมีหลายขั้นตอน
- ต้องการแยกความรับผิดชอบเป็นบทบาทย่อย
- ต้องการ workflow แบบ “วิเคราะห์ → วางแผน → ลงมือ → ทดสอบ → ตรวจทาน”

สรุปสั้น ๆ:

- **rules = policy**
- **skills = capability**
- **agents = execution role**

---

## 3) โครงสร้างที่ดีของ `skills`

ไฟล์ skill ที่ดีควรมีลักษณะดังนี้

- ชัดเจนว่าทำอะไร
- มีขอบเขตงานแน่นอน
- ไม่ยาวเกินจำเป็น
- มีตัวอย่างเมื่อช่วยให้เข้าใจเร็วขึ้น
- มีเงื่อนไขว่า “ใช้เมื่อไร” และ “ไม่ควรใช้เมื่อไร”

### ตัวอย่างที่ดี

- `analysis` — ใช้เมื่อจะทำความเข้าใจโค้ด, dependency, data flow
- `code` — ใช้เมื่อจะ implement หรือแก้โค้ด
- `test` — ใช้เมื่อจะเขียน/รัน/ตรวจสอบ test
- `debug` — ใช้เมื่อมี bug หรือพฤติกรรมผิดปกติ
- `review` — ใช้เมื่อต้องตรวจคุณภาพ, ความปลอดภัย, performance

### หลักการตั้งชื่อ skill

- ใช้ชื่อสั้น จำง่าย สื่อความหมายตรง
- ถ้าเป็นงานเฉพาะมาก ควรแยกเป็น skill เฉพาะทาง
- ถ้าคล้ายกันมาก ให้รวมไว้ด้วยกันเพื่อลดความซับซ้อน

---

## 4) วิธีออกแบบ rule กลางให้ใช้ได้ทุกโปรเจกต์

rule กลางที่ดีควร

1. **ไม่ผูกกับ stack ใด stack หนึ่ง**
2. **ย้ำพฤติกรรมหลักที่ใช้ได้ทุกภาษา**
3. **บอกลำดับความสำคัญในการตัดสินใจ**
4. **หลีกเลี่ยงรายละเอียดที่เฉพาะโปรเจกต์เกินไป**
5. **อ่านแล้วนำไปใช้ได้ทันที**

### ตัวอย่างหัวข้อที่ควรมี

- ความถูกต้องมาก่อนความเร็ว
- อ่านบริบทก่อนแก้
- ยึด convention เดิมของ repo
- ใช้การเปลี่ยนแปลงแบบเล็กและตรวจสอบได้
- หลังแก้ต้องตรวจ lint/test เมื่อมี
- ถ้าไม่มั่นใจ ให้ระบุสมมติฐาน

### หัวข้อที่ไม่ควรใส่ใน rule กลาง

- รายละเอียด port เฉพาะ
- stack เฉพาะ เช่น React, Go, Django, NestJS
- ชื่อโปรเจกต์
- architecture ที่เฉพาะบาง repo เท่านั้น

---

## 5) Workflow ที่แนะนำสำหรับ agent

เวลาจะทำงานในโปรเจกต์ใด ๆ ให้ใช้ลำดับนี้

### ขั้นที่ 1: อ่านบริบท

- อ่านไฟล์ที่เกี่ยวข้อง
- ตรวจ rule ที่มีอยู่
- หา pattern ที่ repo ใช้อยู่จริง

### ขั้นที่ 2: ประเมินงาน

- งานคืออะไร
- มีความเสี่ยงอะไร
- ต้องใช้ skill ไหน
- ต้องใช้ agent บทบาทไหน

### ขั้นที่ 3: วางแผน

- ถ้างานง่าย ให้ทำแบบสั้นและตรง
- ถ้างานใหญ่ ให้แตกเป็นขั้นตอน
- ถ้างานมีความเสี่ยง ให้เพิ่มการตรวจสอบ

### ขั้นที่ 4: ลงมือทำ

- แก้เฉพาะส่วนที่เกี่ยวข้อง
- หลีกเลี่ยงการ refactor เกินจำเป็น
- รักษา behavior เดิมถ้าไม่ได้ตั้งใจเปลี่ยน

### ขั้นที่ 5: ตรวจสอบ

- lint
- test
- diff review
- ความสอดคล้องกับ rule

### ขั้นที่ 6: สรุปผล

- ทำอะไรไป
- ทำไมจึงทำแบบนั้น
- มีความเสี่ยงอะไรค้างอยู่หรือไม่

---

## 6) วิธีเลือก agent ให้เหมาะกับงาน

### `analysis`
ใช้เมื่อ

- ยังไม่เข้าใจ codebase
- ต้อง map dependency
- ต้อง reverse engineer logic เดิม
- ต้องหา hidden business rules

### `plan`
ใช้เมื่อ

- งานมีหลายวิธีทำ
- ต้องตัดสินใจเชิงสถาปัตยกรรม
- ต้องเทียบ trade-off

### `code`
ใช้เมื่อ

- มีแผนชัดแล้ว
- ต้อง implement จริง
- ต้องแก้ไฟล์จำนวนมากหรือน้อยตาม scope

### `test`
ใช้เมื่อ

- ต้องเพิ่ม test
- ต้องรัน suite
- ต้องตรวจ regression

### `debug`
ใช้เมื่อ

- ระบบพัง
- มี bug
- มี error ที่ต้องหาต้นเหตุ

### `review`
ใช้เมื่อ

- งานสำคัญก่อน merge
- ต้องตรวจ performance, security, quality พร้อมกัน

### `docs`
ใช้เมื่อ

- ผู้ใช้ขอ documentation โดยตรง
- ต้องอธิบาย API, workflow, หรือคู่มือ

### `deploy`
ใช้เมื่อ

- พร้อม commit / push
- ต้องตรวจ git state และความปลอดภัย

---

## 7) แนวทางใช้งาน skill และ agent ร่วมกัน

ตัวอย่าง workflow ที่ดี

### งานเล็ก

1. อ่านไฟล์
2. ใช้ `code`
3. ตรวจ `lint` หรือ `test` เฉพาะจุด
4. สรุปผล

### งานกลาง

1. ใช้ `analysis` เพื่อเข้าใจบริบท
2. ใช้ `plan` เพื่อเลือกแนวทาง
3. ใช้ `code` แก้ไข
4. ใช้ `test` ตรวจสอบ
5. ใช้ `review` ถ้างานสำคัญ

### งานใหญ่หรือเสี่ยงสูง

1. `analysis`
2. `plan`
3. `code`
4. `test`
5. `review`
6. `deploy` เมื่อพร้อม

---

## 8) แนวทางเขียน skill ที่ดี

ถ้าจะสร้าง skill ใหม่ ควรมีส่วนเหล่านี้

- **ชื่อ skill**
- **คำอธิบายสั้น ๆ**
- **ใช้เมื่อไร**
- **ไม่ควรใช้เมื่อไร**
- **ขั้นตอนการทำงาน**
- **ข้อควรระวัง**
- **ตัวอย่าง**

### ตัวอย่างโครงสร้าง

- ถ้า skill นั้นเกี่ยวกับการ review ควรระบุว่าจะตรวจอะไรบ้าง
- ถ้า skill นั้นเกี่ยวกับการ debug ควรระบุสัญญาณของ root cause
- ถ้า skill นั้นเกี่ยวกับการ deploy ควรมี safety checks

---

## 9) แนวทางเขียน agent instruction ที่ดี

agent instruction ที่ดีควร

- ระบุบทบาทชัดเจน
- ไม่สั่งกว้างเกินไป
- มีเงื่อนไขการจบงาน
- มีมาตรฐานการตรวจงาน
- ระบุสิ่งที่ห้ามทำ

### ตัวอย่างสิ่งที่ควรมี

- เป้าหมายของ agent
- ขอบเขตไฟล์หรือระบบที่เกี่ยวข้อง
- Acceptance criteria
- สิ่งที่ห้ามแตะ
- วิธีตรวจว่าทำเสร็จแล้ว

---

## 10) มาตรฐานความละเอียดของเอกสารใน `.cursor/`

เอกสารใน `.cursor/` ควรช่วยให้ agent ทำงานได้จริง ไม่ใช่แค่เก็บไว้เฉย ๆ

### ควรทำ

- เขียนให้กระชับแต่ครบ
- ใช้ภาษาที่สั่งงานได้
- ใส่ตัวอย่างเมื่อจำเป็น
- แยกหัวข้อให้ scan ได้ง่าย

### ไม่ควรทำ

- ยาวโดยไม่มีโครงสร้าง
- ข้อมูลซ้ำหลายจุด
- ข้อความกำกวม
- rule ที่ขัดกันเอง

---

## 11) วิธีดูแลและปรับปรุง `.cursor/`

### เมื่อเพิ่ม skill ใหม่

- ตรวจว่ามี skill เดิมใกล้เคียงหรือไม่
- ถ้าซ้ำกัน ให้รวม
- ถ้าเฉพาะทางจริง ให้แยก

### เมื่อเพิ่ม rule ใหม่

- ดูว่าควรเป็น rule กลางหรือ rule เฉพาะไฟล์
- ถ้าผูกกับ stack เดียว ให้ใช้ `globs`
- ถ้าต้องใช้เสมอ ให้ใช้ `alwaysApply`

### เมื่อ agent ทำงานพลาดซ้ำ

- ตรวจว่า instruction กว้างเกินไปหรือไม่
- ตรวจว่า skill ไม่ชัดเจนหรือไม่
- เพิ่มตัวอย่างหรือข้อห้ามเฉพาะจุด

---

## 12) แนวทางสำหรับผู้ดูแล repo

ถ้าต้องดูแล `.cursor/` ให้มีคุณภาพในระยะยาว

- รักษา rule กลางให้ stack-agnostic
- แยก rule เฉพาะเท่าที่จำเป็น
- อัปเดต skill เมื่อ workflow เปลี่ยน
- ลบสิ่งที่ซ้ำซ้อนหรือ obsolete
- ตรวจ consistency ระหว่าง `rules`, `skills`, และพฤติกรรมจริงใน repo

---

## 13) Checklist ก่อนใช้งานจริง

ก่อนใช้ skill หรือ agent ใด ๆ ให้ถามตัวเองว่า

- งานนี้ต้องการมาตรฐานกลางหรือความสามารถเฉพาะทาง
- มีไฟล์ rule ที่บังคับอยู่แล้วหรือไม่
- ต้องอ่าน context อะไรก่อนเริ่ม
- ต้องตรวจผลลัพธ์ด้วยอะไร
- มีความเสี่ยงด้าน logic, quality, หรือ security หรือไม่

---

## 14) สรุปสั้น ๆ

ถ้าจะใช้ Cursor ให้คุ้มที่สุด

- ใช้ `rules` เพื่อคุมมาตรฐาน
- ใช้ `skills` เพื่อเพิ่มความสามารถเฉพาะทาง
- ใช้ `agents` เพื่อแยกบทบาทและเพิ่มประสิทธิภาพ
- ทำงานแบบอ่านบริบทก่อน แก้ทีละส่วน และตรวจผลทุกครั้ง
- รักษาเอกสารใน `.cursor/` ให้สั้น ชัด และนำไปใช้ได้จริง

---

## 15) ข้อเสนอแนะในการต่อยอด

ถ้าต้องการขยายระบบนี้เพิ่มเติม แนะนำสร้างเอกสารแยกเป็น

- `skills-guide.md` — วิธีออกแบบและใช้งาน skills
- `agents-guide.md` — วิธีเลือกและจัดการ agents
- `rules-guide.md` — วิธีเขียน rule กลางและ rule เฉพาะทาง
- `workflow-guide.md` — มาตรฐานการทำงานของ agent แบบ end-to-end
